## Раздел 2. Технические вопросы

## 2.1 Kubernetes

### Вопрос 1. Что такое Kubernetes? Для чего его используют?

**Kubernetes** — это система оркестрации контейнеров. Она используется для автоматического развертывания, масштабирования и управления приложениями, которые работают в контейнерах (например, Docker). Kubernetes помогает запускать приложения на группе серверов (кластере) и следит за тем, чтобы нужное количество экземпляров приложения работало корректно. Если один из контейнеров перестаёт работать, система может автоматически перезапустить его.

### Вопрос 2. Объясните разницу между Pod, Deployment и Service в Kubernetes.

**Pod** — это минимальная единица запуска в Kubernetes. Внутри pod может находиться один или несколько контейнеров с приложением.

**Deployment** управляет pod'ами. Он отвечает за то, сколько экземпляров приложения должно быть запущено и автоматически создаёт или обновляет pod'ы при необходимости.

**Service** обеспечивает доступ к приложению. Он направляет сетевые запросы к нужным pod'ам и позволяет другим сервисам или пользователям обращаться к приложению по стабильному адресу.

**Pods** — это самые маленькие развертываемые единицы в Kubernetes, которые часто сравнивают с одной комнатой в здании, где размещается один или несколько контейнеров, подобно мебели. **Deployment** действуют как управляющий зданием, обеспечивая доступность указанного количества комнат и внося необходимые корректировки. **Service** действуют как секретарь, направляя трафик в соответствующие комнаты и гарантируя, что запросы достигают нужных мест назначения.

### Вопрос 3. Что такое Namespace в Kubernetes? Зачем он нужен?

**Namespace** — это способ логически разделить ресурсы внутри одного Kubernetes-кластера. Он позволяет организовать приложения и сервисы по проектам, командам или средам (например, development, staging, production). С их помощью разные команды могут работать в одном кластере, не мешая друг другу.

### Вопрос 4. Объяснение манифеста Kubernetes
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
 name: my-application
 labels:
     app: my-app
spec:
 replicas: 3
 selector:
     matchLabels:
         app: my-app
 template:
     metadata:
         labels:
             app: my-app
spec:
 containers:
 - name: my-container
 image: my-registry/my-app:v1.2.0
 ports:
 - containerPort: 8080
 resources:
     limits:
     memory: "512Mi"
     cpu: "500m" ```

Манифест описывает **Deployment**, который запускает контейнерное приложение в Kubernetes.  
Он определяет, сколько экземпляров приложения должно запускаться и какие ресурсы должен использовать контейнер.

Ниже приводится объяснение основных разделов.

**apiVersion: apps/v1**  
Указывает, какая версия API Kubernetes должна использоваться для создания этого ресурса.

**kind: Deployment**  
Определяет тип объекта Kubernetes. В данном случае это Deployment, который управляет группой под (pods), запускающих приложение.

**metadata**  
Содержит основную информацию о ресурсе.

- **name: my-application** – имя развертывания.
- **labels** – теги «ключ-значение», которые помогают Kubernetes идентифицировать и организовывать ресурсы.

**spec**  
Описывает желаемое состояние развертывания.

- **replicas: 3** – Kubernetes должен запустить три экземпляра приложения (три пода).

**selector**  
Определяет, как развертывание находит под, которыми оно управляет. Оно ищет под с меткой `app: my-app`.

**template**  
Описывает конфигурацию подов, которые будут созданы.

- **metadata.labels** – метки, назначенные поду.
- **spec** – определяет контейнеры, которые будут запущены внутри пода.

**containers**  
Перечисляет контейнеры, которые будут запущены в каждом поде.

- **name: my-container** – имя контейнера.
- **image: my-registry/my-app:v1.2.0** – образ контейнера, который будет использоваться для запуска приложения.

**ports**  
Указывает, какой порт контейнера

### 2.2. Frontend-разработка
### Вопрос 1. Что такое DOM? Как браузер работает с HTML-страницей?

Document Object Model (DOM) связывает веб-страницы со скриптами или языками программирования, представляя структуру документа — например, HTML-код, описывающий веб-страницу в памяти. Обычно под этим понимается JavaScript, хоть и моделирование документов HTML, SVG или XML в виде объектов не входит в основной набор возможностей языка JavaScript.

DOM представляет документ в виде логического дерева. Каждая ветвь дерева заканчивается узлом, а каждый узел содержит объекты. Методы DOM позволяют программно обращаться к дереву. С их помощью можно изменять структуру, стиль или содержание документа.

К узлам также могут быть привязаны обработчики событий. Как только событие срабатывает, обработчики событий выполняются.

### Вопрос 2. В чём разница между HTML, CSS и JavaScript? За что отвечает каждая технология?

HTML, CSS и JavaScript — это три основные технологии, которые используются для создания веб-страниц.

HTML (HyperText Markup Language) отвечает за структуру страницы. С его помощью описываются элементы страницы, такие как заголовки, абзацы, изображения и ссылки.

CSS (Cascading Style Sheets) отвечает за внешний вид страницы. Он определяет цвета, шрифты, расположение элементов и общий стиль сайта.

JavaScript добавляет интерактивность и динамическое поведение. С его помощью страница может реагировать на действия пользователя, обновлять содержимое и взаимодействовать с сервером.

Часто используют аналогию: **HTML — это «скелет» страницы, CSS — её внешний вид, а JavaScript — логика, которая делает страницу интерактивной.**

### Вопрос 3. Что такое React? Почему он популярен?

React JS — это библиотека JavaScript с открытым исходным кодом, изначально созданная компанией Facebook (ныне Meta) для разработки динамичных пользовательских интерфейсов на основе компонентов.
В отличие от традиционных фреймворков, React позволяет разработчикам разбивать пользовательский интерфейс на повторно используемые компоненты, что ускоряет обновление и упрощает сопровождение сложных приложений.

Виртуальный DOM, ключевая инновация React, оптимизирует рендеринг, обновляя только те части интерфейса, которые изменились, что повышает скорость и эффективность.

Декларативный синтаксис React делает код предсказуемым, упрощает отладку и ускоряет развёртку.
Его компонентная архитектура позволяет командам создавать сложные функции с минимальным дублированием кода. Для разработчиков, знакомых с JavaScript, CSS и HTML, React по-прежнему остаётся одним из самых простых в освоении фреймворков.

Виртуальный DOM и однонаправленный поток данных обеспечивают более быструю прорисовку и стабильность, сокращая количество сбоев приложения и улучшая пользовательский опыт — ключевые факторы, влияющие на SEO-рейтинги и вовлеченность пользователей.

### Вопрос 4. Объясните понятие «компонент» в контексте frontend-разработки. Приведите примеры компонентов на сайте, которым пользуетесь каждый день.

В сфере разработки интерфейсов «компонент» — это готовый к повторному использованию, автономный элемент пользовательского интерфейса, который инкапсулирует свою структуру, поведение и стилевое оформление. Например, «поисковая строка» на веб-сайте, который вы регулярно посещаете, может быть компонентом, равно как и «кнопки», «заголовки» и «меню навигации». Эти компоненты помогают создавать сложные пользовательские интерфейсы, позволяя разработчикам повторно использовать код в разных частях веб-сайта.

### Вопрос 5. Что такое API? Как frontend взаимодействует с backend?

API (интерфейс прикладного программирования) — это набор правил и протоколов для создания программных приложений и взаимодействия с ними. API позволяют интерфейсу пользователя обмениваться данными с серверной частью путем отправки запросов и получения данных. Такое взаимодействие обеспечивает динамическое обновление контента и получение данных, что способствует обеспечению плавного взаимодействия пользователя с веб-приложениями.

Фронт-энд взаимодействует с бэк-эндом через API, отправляя запросы и получая данные для динамического обновления интерфейса. Фронт-энд отправляет запросы в бэк-энд, а бэк-энд обрабатывает эти запросы, выполняя необходимые операции, такие как извлечение или обновление данных, а затем отправляет ответы обратно в фронт-энд. Такое взаимодействие обеспечивает динамическое обновление контента и гарантирует бесперебойную работу пользователей.

### 2.3. Unreal Engine
### Вопрос 1. Что такое Unreal Engine? Для каких задач его используют?

Unreal Engine — это мощный и широко признанный игровой движок, разработанный компанией Epic Games и используемый для создания видеоигр. 

Он предоставляет разработчикам обширный набор инструментов и технологий для создания интерактивной 3D-графики и захватывающих виртуальных миров. Он разрабатывается уже более двух десятилетий и превратился в многофункциональную и универсальную платформу.

### Вопрос 2. Что такое «слой» (Layer) в Unreal Engine? Для чего нужны слои при разработке игры?

Слой (Layer) в Unreal Engine — это способ организовать и группировать объекты (Actors) внутри игровой сцены. Слои помогают разработчикам управлять сложными уровнями, объединяя связанные объекты в отдельные группы.

Например, различные элементы сцены — окружение, персонажи, источники света или здания — можно поместить в разные слои. Затем разработчик может включать или скрывать эти слои в редакторе.

Это позволяет сосредоточиться только на нужной части сцены и упрощает работу с большими игровыми уровнями.

### Вопрос 3. Объясните разницу между Actor и Object в Unreal Engine.

В Unreal Engine **Object** — это базовый тип объекта, который используется для хранения данных или реализации функциональности внутри движка. Такие объекты существуют в памяти, но не имеют позиции в игровом мире.

**Actor** — это специальный тип объекта, который может существовать непосредственно в игровом уровне. Акторы имеют положение, вращение и масштаб, а также могут взаимодействовать с другими объектами сцены.

Проще говоря, **каждый Actor является Object, но не каждый Object является Actor**. Акторы используются для объектов, которые появляются в игровом мире, например персонажей, источников света или камер.

### Вопрос 4. Что такое Blueprint в Unreal Engine? В чём его преимущество перед обычным программированием?

**Blueprint** — это система визуального программирования в Unreal Engine. Она позволяет создавать игровую логику с помощью графических узлов вместо написания кода. Разработчики соединяют узлы в визуальном редакторе, чтобы описать поведение объектов, обработку событий и взаимодействие элементов игры. Главное преимущество Blueprint заключается в том, что он делает разработку более доступной. Дизайнеры и художники могут создавать игровую логику без необходимости писать сложный программный код. Кроме того, Blueprint позволяет быстро создавать прототипы и тестировать игровые механики прямо в редакторе.

## Обзор кластера

Платформа работает на кластере Kubernetes, в котором размещено несколько сервисов приложений. 
Управляющая плоскость Kubernetes доступна по следующему адресу:

https://cluster.example.com:6443

Управляющая плоскость отвечает за управление кластером и координацию запущенных приложений.

Кластер также включает сервис CoreDNS, который обеспечивает внутреннее DNS-разрешение. 
Это позволяет приложениям, работающим в кластере, взаимодействовать друг с другом, используя имена сервисов вместо IP-адресов.

Приложения организованы с использованием пространств имён Kubernetes. Пространства имён позволяют различным средам или командам изолировать свои ресурсы в пределах одного кластера.

В этом кластере доступно несколько пространств имён, в том числе:

- default — пространство имён по умолчанию для ресурсов, в которых не указано пространство имён
- kube-system — содержит основные системные компоненты Kubernetes
- kube-public — используется для общедоступных ресурсов кластера
- kube-node-lease — используется внутренними компонентами Kubernetes
- development — среда, используемая для разработки и тестирования
- staging — среда для предварительного тестирования перед выпуском
- production — среда для работы производственных приложений

## Работа с пространствами имён

В Kubernetes пространства имён используются для организации ресурсов внутри кластера. 
Они позволяют различным средам, командам или приложениям работать независимо друг от друга, используя одну и ту же инфраструктуру кластера.

На данной платформе пространства имён используются для разделения сред, таких как development, staging и production.

### Просмотр пространств имён

Чтобы увидеть список доступных пространств имён в кластере, выполните следующую команду:

```kubectl get namespaces```

Пример вывода:
```NAME            STATUS AGE```
```default         Active 365d```
```kube-node-lease Active 365d```
```kube-public     Active 365d```
```kube-system     Active 365d```
```development     Active 180d```
```production      Active 180d```
```staging         Active 180d```

В столбце `STATUS` отображается, активно ли пространство имён в данный момент.

### Создание пространства имён

Чтобы создать новое пространство имён, воспользуйтесь следующей командой:
```kubectl create namespace <namespace-name>```

Пример:
```kubectl create namespace testing```

После создания пространства имён в него можно развертывать такие ресурсы, как деплойменты, поды и сервисы.

### Выполнение команд в конкретном пространстве имён

Во многих командах Kubernetes можно указать пространство имён с помощью флага `-n`.

Например, чтобы просмотреть поды в пространстве имён `development`:
`kubectl get pods -n development`

Это гарантирует, что команда будет отображать только ресурсы из выбранного пространства имён.

Использование пространств имён помогает разделять различные среды и предотвращает конфликты ресурсов.

## Развертывания в пространстве имён «development»

Пространство имён `development` содержит несколько сервисов приложения, развернутых в виде Kubernetes Deployments. 
Каждое развертывание управляет набором идентичных подов и обеспечивает работу необходимого количества экземпляров.

Чтобы просмотреть развертывания в пространстве имён `development`, выполните команду:
```kubectl get deployments -n development```

Пример вывода:

```NAME                   READY       UP-TO-DATE      AVAILABLE       AGE```
```auth-service           2/2         2               2               45d```
```api-gateway            3/3         3               3               90d```
```user-service           2/2         2               2               60d```
```payment-service        1/1         1               1               30d```
```notification-service   1/1         1               1               15d```

### Обзор развертывания

В настоящее время в пространстве имён `development` работают следующие сервисы:

- **auth-service** — отвечает за аутентификацию и авторизацию пользователей
- **api-gateway** — служит точкой входа для входящих запросов API
- **user-service** — управляет данными и профилями пользователей
- **payment-service** — обрабатывает операции, связанные с оплатой
- **notification-service** — отправляет уведомления пользователям

Вместе эти сервисы формируют архитектуру приложения на основе микросервисов.

---

### Статус развертывания

Колонка `READY` показывает, сколько экземпляров под (pod) в данный момент работает по сравнению с желаемым количеством.

Например:

- `2/2` означает, что из 2 ожидаемых подов запущено 2
- `3/3` означает, что все реплики доступны

Колонка `AVAILABLE` указывает, сколько подов готово обслуживать трафик.

Все развертывания в пространстве имен `development` находятся в исправном состоянии, так как количество доступных подов соответствует желаемой конфигурации.

---

### Сервисы и внутренний доступ

Каждое развертывание предоставляется внутри кластера с помощью сервисов Kubernetes типа `ClusterIP`.

Чтобы просмотреть сервисы, выполните:

`kubectl get services -n development`

Примеры сервисов:

- auth-service — доступен на порту 8080
- api-gateway — доступен на порту 8080
- user-service — доступен на порту 8080
- payment-service — доступен на порту 8080
- notification-service — доступен на порту 8080

Службы типа `ClusterIP` доступны внутри кластера и обеспечивают связь между микросервисами.

Такая конфигурация гарантирует, что службы могут надежно взаимодействовать друг с другом, используя стабильные внутренние конечные точки.

## Просмотр подов и их состояния

Поды — это наименьшие развертываемые единицы в Kubernetes. В каждом поде запускается один или несколько контейнеров. Они представляют собой отдельный экземпляр приложения и отвечают за запуск и управление контейнерами внутри него.

Чтобы просмотреть все поды в пространстве имён `development`, выполните команду:
`kubectl get pods -n development`

Пример вывода:

```NAME                                   READY   STATUS  RESTARTS    AGE```
```auth-service-5f8d9c7b4d-x7k2m          1/1     Running 0           2d```
```auth-service-5f8d9c7b4d-p9n3q          1/1     Running 0           2d```
```api-gateway-6c7b8d9f5c-m4k8p           1/1     Running 0           5d```
```api-gateway-6c7b8d9f5c-n2w7r           1/1     Running 0           5d```
```api-gateway-6c7b8d9f5c-q1t5s           1/1     Running 0           5d```
```user-service-7d8e9f0a6d-k3j6h          1/1     Running 0           1d```
```user-service-7d8e9f0a6d-l5m9n          1/1     Running 0           1d```
```payment-service-8e9f0a1b7e-o2p4q       1/1     Running 0           12h```
```notification-service-9f0a1b2c8f-r3s5t  1/1     Running 0           6h```

### Значения статусов подов

- **READY** — показывает, сколько контейнеров в под запущено по сравнению с общим количеством, указанным в спецификации пода.
- **STATUS** — текущее состояние пода (например, Running, Pending, CrashLoopBackOff).
- **RESTARTS** — количество перезапусков пода с момента создания.
- **AGE** — время работы пода.

Например:

- `1/1 Running` означает, что под имеет один контейнер, который успешно работает.
- Под со статусом `0/1` указывает, что контейнер не готов.

Мониторинг статуса подов позволяет убедиться, что приложения работают корректно, и быстро выявлять проблемы.

### Просмотр логов и устранение неполадок

Логи содержат подробную информацию о работе ваших приложений и играют важную роль при устранении неполадок в кластере Kubernetes. Поды генерируют логи контейнеров, что позволяет отслеживать работу приложений и выявлять проблемы.

Чтобы просмотреть логи для конкретного пода в пространстве имён `development`, воспользуйтесь следующей командой:
`kubectl logs <pod-name> -n development`

Пример:
`kubectl logs auth-service-5f8d9c7b4d-x7k2m -n development --tail=20`

Эта команда отображает последние 20 строк логов пода auth-service.

### Виды логов

**INFO** — информационные сообщения о нормальной работе (например, запуск сервера, загрузка конфигурации).

**ERROR** — указывает на возникновение проблемы, например, сбой аутентификации или ошибки подключения к базе данных.

**Логи HTTP** — отображают входящие запросы, коды статуса и время отклика (например, POST /api/v1/login - 200 OK - 45 мс).

Мониторинг логов позволяет убедиться, что приложения запускаются правильно, обрабатывают запросы в соответствии с ожиданиями, а также быстро обнаруживать ошибки.

### Распространенные проблемы и решения

1. Под не запускается
- Проверьте состояние пода с помощью команды `kubectl get pods -n <namespace>`.
- Просмотрите логи с помощью команды `kubectl logs <имя_пода>`.
- Убедитесь, что образ контейнера существует и конфигурация верна.

2. Приложение не может подключиться к базе данных
- Убедитесь, что служба базы данных работает и доступна внутри кластера.
- Проверьте переменные среды и строки подключения в конфигурации развертывания.

3. HTTP-запросы возвращают ошибки
- Используйте логи, чтобы определить, какая служба или конечная точка вызывает проблему.
- Убедитесь, что все зависимые службы доступны и правильно настроены.
